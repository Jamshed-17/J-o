## 1. Дихотомия сущностей: Сильные vs Слабые

В теории баз данных сущности разделяются по принципу их идентификационной независимости:

- **Правильные (сильные) сущности**: Обладают **атрибутивным самодетерминизмом**. Это значит, что их экземпляр можно однозначно идентифицировать с помощью собственного первичного ключа без обращения к другим объектам системы.
    
- **Слабые сущности**: Характеризуются **зависимостью по существованию**. Они не имеют собственного ключа, достаточного для идентификации. Их существование логически невозможно без родительской (сильной) сущности (например, «Аккорд» в контексте «Песни»).
    

## 2. Модальность связей: Полное и частичное участие

Этот параметр определяет обязательность присутствия экземпляра сущности в связи:

- **Полное участие (Total Participation)**: Экзистенциальная зависимость, при которой _каждый_ экземпляр сущности **обязан** быть связан с экземпляром другой сущности. На диаграммах часто обозначается двойной линией.
    
- **Частичное участие (Partial Participation)**: Опциональная связь. Допускается существование экземпляров сущности, которые не задействованы в данной связи.
    

## 3. Нотация Питера Чена (Chen Notation)

Классический академический стандарт визуализации:

- **Сущности**: Прямоугольники.
    
- **Связи**: Ромбы, соединяющие сущности.
    
- **Атрибуты**: Эллипсы, присоединенные к сущностям.
    
- **Кардинальность**: Указывается символами над линиями ($1$, $N$, $M$).
    

## 4. Нотация Ричарда Баркера (Barker's Notation)

Более прикладной стандарт, используемый в продуктах Oracle:

- **Сущности**: Прямоугольники с закругленными углами («soft boxes»).
    
- **Связи**: Линии, где **пунктир** означает частичное участие (может быть), а **сплошная линия** — полное (должен быть).
    
- **Кардинальность**: Используется «воронья лапка» (crow's foot) для обозначения множественности ($N$).
    

## 5. Декомпозиция связи «многие-ко-многим» (M:N)

В реляционной модели прямая реализация связи $M:N$ невозможна. Для её разрешения проводится **нормализация**:

1. Создается дополнительная **ассоциативная сущность** (связующая таблица).
    
2. Связь $M:N$ расщепляется на две связи типа $1:N$.
    
3. Первичный ключ новой таблицы обычно является составным и включает внешние ключи (FK) обеих исходных таблиц.
    

## 6. Нотации IDEF1X и IE в среде ERwin

- **IDEF1X**: Строгий стандарт, ориентированный на логическое моделирование. Различает идентифицирующие (сплошная линия) и неидентифицирующие (пунктир) связи.
    
- **IE (Information Engineering)**: Использует «вороньи лапки». Она более наглядна для визуального анализа мощности связей.
    

## 7. Расширение модели (Проектирование)

При добавлении сущностей «Пользователь» и «Группа пользователей» реализуется связь $M:N$ (один пользователь в разных группах), что требует создания таблицы-перемычки. Для «Периферийных устройств» применяется **супертип** (устройство) и **подтипы** (принтер, сканер) для реализации механизма наследования атрибутов.

## 8. Переход к физической модели

Физическая модель учитывает специфику конкретной СУБД (PostgreSQL, Oracle, MySQL):

- Логические атрибуты превращаются в типы данных (VARCHAR, INTEGER).
    
- Создаются **Представления (Views)**: Виртуальные таблицы для упрощения доступа к данным (например, `vw_user_computers`).
    

## 9. Прямое проектирование (Forward Engineering)

Это процесс генерации **DDL-скрипта** (Data Definition Language) из графической модели. ERwin или аналогичный инструмент автоматически формирует SQL-запросы (`CREATE TABLE`, `ALTER TABLE`) для создания структуры базы данных непосредственно на сервере.
